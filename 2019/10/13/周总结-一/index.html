Object
Object:  


概述：Object是类层次结构的根类。是所有其他类的父类。所有对象（包括数组）都实现这个类的方法。
构造方法：


Object();
Object类中没有任何属性
这也是为什么默认会提供无参的构造方法。也是子类会访问父类的无参构造方法的原因。  
方法：


protected Object clone()
poublic boolean equals(Object obj)
protected void finalize()
Class<?> getClass()  // 获取对象的运行时类型信息
int hashCode()
String toString()
Object toString方法：
public String toString():


概述：返回该对象的字符串表示。结果应是一个简明但易于读懂的信息表达式。建议所有子类都重写此方法。
Object中的实现: return getClass().getName() + "@" + Integer.toHexString(hashCode());
注意事项  


a. 直接打印对象, 默认会调用该对象的toString()方法。例外：char[]例外。
b. 需要重写toString()，返回该对象的简要信息。一般根据关键属性重写toString()方法。
Object equals方法：
public boolean equals(Object obj)：（注意继承关系中的equals方法的书写，视情况定）


概述：判断 this 是否和 obj 是否"相等"。
约束：
    自反性：对于任何非空引用值 x，x.equals(x) 都应返回 true。
    对称性：对于任何非空引用值 x 和 y，当且仅当 y.equals(x) 返回 true 时，x.equals(y) 才应返回 true。
    传递性：对于任何非空引用值 x、y 和 z，如果 x.equals(y) 返回 true，并且 y.equals(z) 返回 true，那么 x.equals(z) 应返回 true。
    一致性：对于任何非空引用值 x 和 y，多次调用 x.equals(y) 始终返回 true 或始终返回 false，前提是对象上 equals 比较中所用的信息没有被修改。
    非空性：对于任何非空引用值 x，x.equals(null) 都应返回 false。
Object hashCode方法：
public int hashCode():


概述：返回该对象的哈希码值。本质是一个hash算法。
约束：
一致性：一次运行期间, 如果equals比较的属性没有发生修改, 多次调用hashcode方法，应该返回相同的整数。
如果两个对象“相等”，那么它们的 hashcode() 返回的整数应该相等。
尽量为“不相等”的对象，生成不同的整数结果。
Object中hashcode()的实现：


将内存地址映射程一个整数。不同的内存地址映射成不同的整数。和Object中equals一样，都是判断是否是同一个对象。
注意事项：


如果重写了equals方法，一定要重写hashcode()方法。而且要根据equals比较的属性生成hashcode().
Object finalize方法：
protected void finalize():


概述：
当垃圾器回收该对象时，会由垃圾回收器调用该方法。子类重写 finalize 方法，以配置系统资源或执行其他清除。
Object中的默认实现: 空实现。
注意事项：


a. 直接调用相当于普通的方法调用,该对象不会变成垃圾。
b. 一般回收资源我们不会放在该方法中, try...finally子句中。

String
public final class String implements Comparable:


final: String类型的引用指向得对象, 一定是String类型得对象。
概述：String 类代表字符串。Java 程序中的所有字符串字面值都作为此类的实例实现。
字符串是常量；它们的值在创建之后不能更改。因为 String 对象是不可变的，所以可以共享。

Tips:

第一种在栈中new了一个新的对象，第二种是直接从常量池中直接获取；

Exception
体系结构：


Throwable:  
|-- Error  
|-- Exception  
    |-- 编译期异常  
     |-- 运行时异常(RuntimeException 及其子类)
JVM对异常的默认处理： 


a. 打印出现问题的线程名称  
b. 异常的类型  
c. 异常的信息 
d. 异常的调用栈
e. 中断程序



注意事项：
    try块里面的代码一旦出现异常，异常后面的代码不会被执行。
异常处理的机制：
    如果try语句块里面出现了异常, JVM就会把异常封装成对应类型的对象，然后抛出去.
    然后依次和catch语句进行匹配, 如果是对应的异常类型，就执行对应的处理。


JDK7的新特性：一个catch语句可以处理多个类型的异常。这个异常之间没有子父类关系
    catch(ExceptionA | ExceptionB e) {


编译期异常和运行时异常的区别： 


编译期异常（受检查异常）：不管会不会发现异常，都必须检查并处理, 否则编译不通过。

运行时异常：
运行时异常(不受检查异常): 不管会不会发现异常，可以处理也可以不处理，编译都能通过。
推荐不处理，一般出现运行时异常，都是代码逻辑有问题, 应该修改逻辑。
throws 和 throw 的区别： 


throws
用在方法声明上，跟的是异常类名, 可以有多个异常。表示可能会抛出哪些异常。
如果出现了异常，由调用者处理。

throw
用在方法内，跟的是异常对象，只能抛出一个异常。确确实实抛出了一个异常。
表示抛出异常，由方法体内的语句处理
final,finally和finalize的区别: 


a. final是一个关键字.
    变量：
        基本数据性：变量的值不可改变
        引用数据类型：指向同一个对象, 对象可以改变。
    方法：方法不能被重写
    类：类不能被继承。
b. finally
    try的子句。不管有没有异常，finally里面的代码都会执行。一般使用释放资源的。
c. finalize
   定义在Object里面的方法，
   特点：垃圾回收期回收该对象的时候，会主动调用该方法。优点类似C++析构函数, 设计的初衷是用于释放资源的。
   但是由于一些问题, 现在一般释放资源都放于finally语句中。
Collection
数组和集合都是容器, 那为什么有了数组，还需要集合呢？


a. 长度固定的
b. 只能存储同一种类型的数据
c. 数组没有提供方便操作的方法

集合：
a. 长度是可变的
b. 存储多种类型的数据
c. 集合提供方便操作的方法

集合的弊端：
a. 只能存储引用数据类型
b. 效率没有数组高。
Collection 层次结构 中的根接口。
    一些 collection 允许有重复的元素，而另一些则不允许。一些 collection 是有序的，而另一些则是无序的.




JDK5后的特性：自动装箱和自动拆箱：


装箱：基本数据类型 --> 引用数据类型
拆箱：引用数据类型 --> 基本数据类型
byte       Byte
short      Short
int        Integer
long       Long
char       Character
float      Float
double     Double
boolean    Boolean
